## 重点检查

> 本阶段只检查"叙事中间稿"层面的质量，不检查镜头级字段和分镜级秒数闭合。若剧本阶段在用户明确要求下写了**场次级时长预算**，本命令只检查场次级外置算术、禁止含糊话术和正文负载是否能撑住时长；镜头级问题一律推迟到 `/分镜质检`。

- 名字、称谓是否统一
- 台词是否符合当前人设
- 是否乱加功能性角色
- 相邻集是否承接
- 因果和代价是否成立
- **情节点落地检查**：每集剧本初稿必须能在正文中定位到 `分集大纲.md` 的"本集情节点"和"情节点后果"；允许为了戏剧效果调整呈现顺序，但不能替换成新的主事件或只保留标题级对应。若正文没有实际发生该情节点 = P1；关键集/付费卡点/结局集偏离大纲情节点 = P0，并回退同步 `分集大纲.md` 或重写本集剧本
- 单集场次数量是否匹配 60-90 秒项目的宏观篇幅参考（推荐 2-5 场）
- 每场是否都能说清承接点、冲突目标、事实锚点、**场景状态差量**和情绪位置（场景状态差量字段权威定义见 `references/episode-script-templates.md` §模板字段说明）
- **场景状态差量逐场扫描**：每场必须有"开场状态 → 收场状态"的具体变化；写"无变化"或留空 = 灌水场，判 P1，要求合并或删除
- **写前检查留痕（强制 P0）**：每集剧本文件头部**必须**包含 `references/episode-writing-protocol.md` §二的 A/B 组 14 问 + C 组 7 项 + D 组 5 项完整答案 HTML 注释；§二 是问题全文唯一权威源，质检只按编号 1-26 与答案完整度判定，不按命令文件或模板里的示例措辞判定。A/B 组注释缺失、残缺或任一问留空 = **P0**，直接阻塞。注释完整后抽查 B 组（问题 9-14）是否真的对应到剧本结构；若 B 组答案虚化且本批次出现"灌水场 / 无尾钩拉力 / 无记忆点 / 信息差停滞 / 时长口头闭合" = P1
- 是否存在只为占篇幅而保留的解释场、空转场、重复反应场
- **短剧观众体验 C 组检查**：逐集检查写前自检是否包含 C 组 15-21 项（肉眼冲突、可视化打脸、女主情绪外化、尾钩具象、双线回击、时长负载、可信度/术语转译）。C 组整块缺失 = P0；单项缺失或虚化 = P1，关键集/付费卡点/证据法庭推进集升级为 P0
- **密度与表现 D 组检查**：逐集检查写前自检是否包含 D 组 22-26 项（三密度、节奏锚、钩子与拉扯、角色高光/反差、展示式表达）。D 组整块缺失 = P1；第 1 集、关键集、付费卡点或重大揭露集缺失 = P0。D 组任一项只写空泛判断（如“节奏快”“有钩子”“有画面感”）= P1
- **三密度正文对齐检查**：剧本正文必须能实际支撑 D 组写下的情绪密度、信息密度、情节密度。若写前检查写了三密度，但正文没有新信息、不可逆变化或情绪释放，判 P1；关键集/付费集 = P0
- **节奏锚正文对齐检查**：第 1 集、关键集、付费卡点集、重大揭露集必须在正文中落地开头冲击、15 秒变化、45 秒强期待、结尾钩子。缺 2 项以上 = P0；普通集没有中段变化 = P1
- **展示式表达检查**：关键身份、威胁、夺权、压迫或反杀若主要靠自报家门式台词成立，而非动作、道具、站位、旁人反应或可见后果，判 P1；主角/反派首次高光这么写 = P0
- **主角反差/出场检查**：主角或核心反派首次关键亮相必须改写现场权力关系或情绪场。若只是“走进来 + 自我介绍 + 众人说明其身份”，判 P1；被结构标为高光/卡点时升 P0
- **可视化打脸/回报检查**：本集正文必须有被打脸者、见证人、受益人或明确反应方。若只看到“文件被提交/法官采纳/申请通过/证据归档”，但没有人物反应或可见代价，爽点判弱，P1；关键集/卡点集 = P0
- **编号式尾钩检查**：结尾钩子若只出现 `Exhibit N`、`Phase N`、`Plan B`、`USB`、`某文件` 等符号，未交代具体兴趣点（内容/危险/倒霉对象/下一集冲突），判 P1；付费卡点 = P0
- **藏线落地检查**：若本集大纲标记了藏线 ID，正文是否用动作、道具、反应、系统字卡或场景异常落地发现证据；若只靠解释台词或根本没有可回看的证据 = P1，关键集/付费卡点 = P0。若为了藏线导致本集显性危机、主角目标或状态变化不清楚 = P0
- **藏线兑现节奏检查**：检查 `分集追踪.md` 与本批次剧本，核心藏线是否在 5-8 集内有小回报、8-12 集内有反向理解或部分兑现；超期无回报 = P1，主线核心藏线超期且影响追更 = P0
- **证据获取阻力检查**：证据、线索、申请、口供、文件不能“一要就有”。正文需出现至少一种阻力或代价：反派阻挠、合法性风险、交换条件、暴露风险、盟友分歧、时间压力、证据缺口。连续 2 集无阻力获取证据 = P1
- **法庭/程序可信度检查**：法律项目不得默认“一份证据开一次庭”。若本集写法庭/预审/提交，必须在场景事实锚点里说明它属于正式庭审、预审动议、同一庭审连续呈堂、庭外提交还是行政程序；若程序场景只是为了方便打证据且连续出现，P1，严重时 P0
- **专业术语转译检查**：每个关键专业术语或程序动作必须在正文或元信息中转译为观众后果。长段法条/专业对白超过 2 行且没有“这意味着谁完了”的短句或反应画面 = P1
- **女主情绪代入检查**：冷静型女主可以不哭不吼，但必须有可视化、可 AI 生成的情绪外化。连续 3 集只有冷静策略、短句和无表情，没有身体代价、动作迟疑、关系冲突、选择痛感或他人点破 = P1
- **双时间线碰撞检查**：含过去线/闪回的项目，过去线不得只做背景解释。若本集有过去线，必须检查同集或下一集是否有同源现在线回击；没有则 P1，关键集 P0
- **正文负载检查**：场次时长外置算术闭合不代表正文能撑 75 秒。若本集主要由电话、独处、文件交接、沉默看物构成，且没有 2 个以上表演节拍/冲突转折/反应镜头，判 P1，要求并场或加压
- **现实可信度检查**：现代现实题材中出现“12 天没睡”等绝对化生理表达、无解释的快速开庭/判决/取证、过度便利的跨境资料获取，判 P1；若该表达支撑关键剧情且不修会破坏可信度，升 P0
- **角色实体锁检查**：亲属关系、墓碑、案名、旧文件、签字页、照片背面、信件称谓必须与 `故事设定.md` 标准名/别名边界一致。出现父亲/母亲/配偶等身份名漂移 = P0；若存在旧姓/别名但设定未定义边界，也按 P0 回退设定
- **反向检查**：是否误写了分镜级字段（镜头号、镜头级时长求和、`AI画面提示`、`AI动作提示`、机位/运镜/景别等）
- **本集调用资产清单**是否已登记本集出场的角色/场景/道具，以及本集新增/变化的长期事实
- 本批次中每个出场的核心角色，其"主要戏剧功能"是否被实际调用
- 检查 `分集追踪.md` 悬念债务表：当前挂起超过 8 集且无部分兑现的悬念 = P1，需在后续 3 集内安排部分兑现；超过 12 集未处理升为 P0
- 本批次中角色重大决定是否与 `故事设定.md` 角色档案的隐性伤口/情绪触发器/失控点一致
- 奇幻/超自然题材：涉及超常能力的场景是否均符合 `故事设定.md` 能力边界表的"正常上限/触发条件/触发代价"
- 核心角色成长弧线锚点事件是否按时发生或正在推进
- **台词个性化（clean-context 盲测优先）**：默认通过当前运行环境可用且已获授权的独立评审通道（例如子 agent、独立会话或外部评审 agent）做盲测判别，详见本文件 §盲测执行规范。若当前运行环境、工具规则或用户授权状态禁止独立评审，执行 §盲测执行规范的"执行环境受限降级规则"：不得伪称 clean-context，只能做本地遮名风险筛查，并在 `质检检查点.md` 与 `qcStatus.episodes.notes` 中明确记录 `clean-context 未执行 / 待补测`。
- 本批次是否至少给核心角色留下可被观众记住的动作、表情、场面或关系翻页点
- **核心角色高光分配**：对本批次每个出场的核心角色（`故事设定.md` 有完整档案的角色），检查是否至少有 ≥1 个"该角色是焦点"的时刻——主动驱动动作（非被动反应）、揭示信息（非被告知）、或做出影响剧情的决定。功能性角色（法警、前台等）不要求。连续一个批次某核心角色只做反应器 = P1
- 本集调用资产清单中出现的角色/场景/道具是否都能在 `故事设定.md` 找到对应记录（`AI资产圣经.md` 首次创建延后到 `/分镜脚本`）

## 阻塞条件

- 角色口吻失真
- 相邻集断裂
- 情节强推
- 关键集/付费卡点/结局集没有落地 `分集大纲.md` 的本集情节点和情节点后果
- 剧本中误写镜头号、镜头级时长求和、AI 画面/动作提示、机位/运镜/景别等分镜级字段（P0）
- **写前检查留痕缺失**：剧本文件头未包含完整 A/B 组 14 问答案 HTML 注释，或任一问留空 / 整块缺失（P0）；C 组整块缺失（P0）；D 组在第 1 集、关键集、付费卡点或重大揭露集中缺失（P0）
- **YAML frontmatter 缺失**：剧本文件头未包含 `episode / version / updated_at` 三字段 frontmatter，或 `version` 为 0 / 非整数（P0）
- 场景只是为了占篇幅存在
- 关键事实、冲突或尾钩被埋在低效场景里，本集结构失焦
- 新角色冗余且未登记
- 核心角色连续 5 集以上未调用其主要戏剧功能
- 悬念挂起超过 12 集且无任何部分兑现
- 角色重大决定存在明显动机断层且无前期铺垫
- 奇幻题材出现无代价突破能力上限的开挂
- 连续一个批次只有功能推进，没有任何角色高光或关系记忆点
- 连续一个批次主要靠证据编号/程序提交推进，缺少直观打脸、可见代价或人物反应
- 关键集/付费卡点没有具象尾钩，只用编号、代号、文件名或抽象阶段名收尾
- 本集大纲标记的核心藏点没有正文证据，或关键真相揭露完全靠突然解释、前文无法回看确认
- 第 1 集前 5 秒仍是背景说明、世界观解释或慢描环境，没有施压/被压/危机动作
- 关键集/付费卡点集的 D 组缺失或完全虚化，导致三密度、节奏锚、钩子拉扯无法在正文中定位
- 法庭/程序场景严重便利化，形成“一份证据开一次庭/交一次材料立刻采纳”的主结构，且无戏剧化压缩说明或替代场景设计
- 双时间线项目在关键集只铺过去伤害，没有同集或下一集的现在线回击设计
- 角色硬身份漂移，例如同一父亲在墓碑、案名、旧文件、设定中出现互斥姓名且无设定解释
- 本集调用资产清单缺失，或与 `故事设定.md` 长期不一致
- **台词个性化 clean-context 盲测准确率 <80%**（P0；主 agent 本地遮名结果一律不得冒充 clean-context。若当前执行环境无法派发独立评审，不能写"clean-context 通过"，必须按"执行环境受限降级规则"留痕；本地筛查若发现明显同质化或声线漂移，按 P1/P0 修源）

## 盲测执行规范（clean-context 独立评审）

> 本命令的核心独立信号是 clean-context 盲测——由一次全新上下文的独立评审通道做"遮名可辨"判别。**原 `episode-writing-protocol.md` §五的"每场写完自检"仍保留作为写作阶段参考**，但进入 `/剧本质检` 的**权威判定**以本规范为准，不以主 agent 自测为准。写剧本的 LLM 自测必然偏袒自己写的台词，信号不可用。

### 为什么必须 clean-context

- 主 agent 当前上下文已携带：本集大纲、状态差量、写前检查、创作意图——这些信息会让它"脑补"出每句台词属于谁，而非真的凭声音判别
- clean-context 独立评审上下文没有任何创作意图包袱，只能凭角色声音档案和台词本身匹配——这才是观众实际的判别条件
- 独立信号是打破 self-reward hacking 的唯一手段

### 执行环境受限降级规则

有些 Codex / Agent 运行环境会禁止在用户未明确授权时启动独立评审通道。此时优先级高于本文件的默认盲测要求，按以下规则执行：

1. **不得绕规则启动独立评审**：若上级工具规则或用户授权状态不允许派发独立评审，不得自行启动，也不得把本地评估包装成"独立盲测"。
2. **本地遮名只作风险筛查**：仍需按本节的抽样、grep 回源、单词级台词剔除、archetype 1:1 映射规则制作样本；主 agent 可做逐行声线复核，但输出只能叫"本地遮名风险筛查"，不能叫 clean-context。
3. **留痕必须透明**：`质检检查点.md` 必须写明 `clean-context 未执行，原因：当前执行环境/授权状态禁止独立评审`，并列出本地样本、行号、判断依据和待补测说明；`qcStatus.episodes.notes` 同步写 `clean-context 待补测`。
4. **发现问题照常阻塞**：本地筛查若发现角色声线明显同质化、关键角色无法区分或用户已明确指出台词问题，仍按 P1/P0 处理并修源。
5. **严格闸门项目需补测**：若用户要求"严格 clean-context 通过后才能推进"，或项目处于导出/交付前，总检前必须让用户明确授权独立评审通道或安排另一个独立会话补跑盲测。

### 执行步骤

1. **抽样**（由主 agent 完成）
   - 从本批次每一集抽取 2 场主要对话：含 ≥2 个核心角色、≥6 轮对白
   - 若单集主要对话不足 2 场，全部取
   - **优先抽**：冲突场、情绪峰值场、关系翻页场；**禁止**避开高密度对话场
   - 单次盲测目标：覆盖本批次全部核心角色至少一次

2. **构造盲测输入包**（由主 agent 完成，严格遵守"最小信息"原则）
   - **声音档案**：从 `故事设定.md` §1.10 角色台词声线卡抽出本批次涉及的核心角色"说话声音档案"四字段（句式偏好 / 词汇禁区 / 情绪表达方式 / 信息处理方式），每个角色不超过 150 字。§1.10 缺失或核心角色四字段不全 = P0，回退补齐后再质检
   - **匿名对白**：把抽样场次的所有角色名与称谓替换为 `角色A / 角色B / 角色C...`，保留动作行（`△`）但去掉任何身份泄露词（如"总裁""医生"等标签称谓也要匿名化）
   - **禁止携带**：本集大纲、状态差量、写前检查、创作意图、情绪曲线、前情提要、尾钩——独立评审方**只能**看到声音档案 + 匿名对白
   - **映射表**：主 agent 保留一份 `角色A → 真实角色名` 的映射表本地留底，供评分用，**不**发给独立评审方

3. **派发独立评审**（使用当前运行环境可用且已获授权的独立上下文；例如子 agent、独立会话或外部评审 agent；若当前运行环境禁止独立评审，跳过本步并执行"执行环境受限降级规则"）

   Prompt 模板：
   ```
   你是独立戏剧评审员。任务：凭角色声音档案判别匿名对白的说话人。

   【规则】
   - 只凭"声音档案四字段"和"台词本身"判断，不要推理剧情
   - 每一句对白给出你认为最匹配的匿名标签（角色A/B/C...）
   - 每条判断必须说明依据：声音档案的哪条字段支撑
   - 不要假设任何未提供的信息，不要脑补上下文

   【声音档案】
   {四字段表}

   【待判别对白】
   {匿名化对白，每句必须包含 line_id、source_line_number、utterance}

   【输出格式】
   JSON 数组：[{line_id, source_line_number, utterance, predicted_speaker, evidence_field, confidence(0-1)}, ...]
   ```

4. **评分**（主 agent 用本地映射表对比）
   - 分母 = 独立评审实际判别的对白总句数
   - 分子 = `predicted_speaker` 与真实角色匹配的句数
   - **准确率 < 80%** → P0 阻塞，判定本批次存在"同质化台词"，写 `qcStatus.episodes[{range: 起止集, checkType: "script"}].status = 需修改`
   - **80% ≤ 准确率 < 90%** → P1，列出独立评审判错的具体场次和句子，要求在本批次内修正后重跑盲测
   - **准确率 ≥ 90%** → 台词个性化检查通过

5. **留痕**（写入 `质检检查点.md`）
   - 抽样场次清单
   - 声音档案片段（原样复制）
   - 独立评审完整回复（JSON）
   - 映射表与命中率
   - 被判错的对白行号与修复建议
   - **禁止**只写"已做盲测"或"盲测通过"——无证据等同未做

### 样本质量硬约束（QC-015.b 沉淀 · 2026-04-12）

> **背景**：E1-E3 Round 1-3 盲测出现两类"伪信号"——L02 主 agent 编造了源剧本里不存在的台词行（伪造样本），L15 把"Yes."这种单词级回答纳入盲测（信号量不足）。Round 3 为此剔除 L02+L15 才把有效准确率从 88.24% 修正到 100%。本节把这两类漏洞写成硬规则，防止后续批次重犯。

**主 agent 抽样时必须遵守**：

1. **台词必须 grep 回源**：每一条纳入盲测的台词，主 agent 必须用 Grep 工具从 `分集剧本/第NNN集.md` 原文读出来，保留 `{文件名, 行号, 场次 ID, 角色名}` 四字段锚点。不允许凭记忆复述、也不允许"差不多是这句"。伪造样本 = 质检直接 P0，本批次 `qcStatus.episodes[{range: 起止集, checkType: "script"}].status = 需修改`。
2. **单词级台词禁入盲测**：≤2 个英文单词（或中文 ≤4 字）的对白不纳入盲测样本——如 "Yes."、"No."、"I know."、"知道了"、"嗯"——这些句子任何角色都能说，信号量 = 0。抽样时自动跳过，改抽同场次下一条 ≥3 词/5 字的台词。
3. **archetype 定义必须随 prompt 下发**：主 agent 构造盲测 prompt 时，必须把本轮全部 archetype 和它们对应的匿名化声音档案一次性写进 prompt 模板，不允许独立评审方自行"发现" archetype。archetype 命名空间按核心角色数量从 `A-F` 依次取用；若本轮核心角色超过 6 个，拆成多轮盲测，每轮不超过 6 个 archetype。不得使用中文/职业词/性别词。
4. **archetype ↔ 核心角色 1:1 映射**：一次盲测的 archetype 数量必须等于本批次"有≥6 轮对白"的核心角色数量，严格 1:1。不得出现"把两个角色合并成一个 archetype"或"一个角色分裂成两个 archetype"。映射表主 agent 本地留底，不下发独立评审方。
5. **盲测 prompt 必须包含 `source_line_number` 字段**：独立评审返回的 JSON 每一条必须携带 `source_line_number`（与主 agent 抽样时锚定的行号一致），供主 agent 事后与本地映射表对照判分。缺失 `source_line_number` = 独立评审回复无效，重抽重跑。
6. **功能性角色不纳入 archetype**：法警、陪审团成员、法官、前台、接待员、助理、服务员等"功能性群众角色"的对白不进盲测样本，因为它们本来就没有设计独立声音档案。只有 `故事设定.md` 明确给出"说话声音档案四字段"的角色才能成为 archetype。

**独立评审回复后主 agent 必须做的校验**：

- 把独立评审 JSON 的每条 `source_line_number` 与主 agent 本地抽样时记录的行号逐行对照。对不上 = 独立评审脑补出了样本外的行号，整包作废，重跑。
- 把独立评审判别的每条台词原文与 `分集剧本/第NNN集.md` 对应行号实际文本做字符级比对。不一致 = 主 agent 抽样时有伪造嫌疑，本批次 P0。

### 禁止清单（违反即判 P0）

- 主 agent 自己在当前上下文内"盖住角色名"并声称 clean-context / 独立盲测通过——零信号价值；本地遮名风险筛查必须明确标为非 clean-context
- 把完整 `故事设定.md` 或角色档案全文塞给独立评审方（超出四字段就是信息泄露）
- 把本集大纲/状态差量/尾钩/写前检查塞给独立评审方（剧情脑补污染）
- 抽样时避开冲突场、情绪峰值场，只抽平淡对话凑数
- 用一次抽样覆盖不到本批次全部核心角色（核心角色缺席 = 盲测无效）
- **抽样时凭记忆复述台词或编造源剧本里不存在的行**（QC-015.b · L02 伪造样本 = 直接 P0）
- **把 ≤2 英文单词 / ≤4 中文字的对白纳入盲测样本**（QC-015.b · L15 单词歧义 = 样本无效，重抽）
- **独立评审回复缺失 `source_line_number` 字段或与本地锚点行号对不上**（整包作废，重跑）
- **把功能性角色（法警/陪审团/法官/前台/服务员）纳入 archetype 命名空间**（archetype 必须对应有声音档案四字段的核心角色）

### 与 `episode-writing-protocol.md` §五的关系

- `episode-writing-protocol.md` §五定义了**写作阶段**的台词个性化要求（声音档案四字段 + 每场写完自检 3 问）
- 本规范定义了**质检阶段**的独立评估机制（clean-context 独立评审盲测）
- 两者不冲突：前者是"写前看"，后者是"写后验"。质检阶段的权威判定以本规范盲测结果为准。

## 修源闭环规则（E7-E9 沉淀 · 2026-04-12）

> **边界说明**：`/剧本质检` 的扫描阶段只读主流程产物，只负责发现问题、写 `质检检查点.md`、更新 `qcStatus.episodes`。本节规则只在后续进入"修复本批次剧本"动作时生效；修复动作可以由用户明确要求，也可以由当前会话在质检结论后继续执行，但必须先把问题 ID 记入 `质检检查点.md`，再按本节顺序修源。

> **背景**：`/剧本质检 E7-E9` 批次发现三类"修源不闭环"踩坑：
>
> 1. **口头 PASS 零验证**：生成 agent 写完场次时长区间后声称 "总计约 75s 覆盖目标"，但外置 Python 加法穿底（E7 56-69s / E8 58-72s 缺口 6-19s）。修源必须走外置算术，不走主 agent 口头自检。
> 2. **outline vs script 漂移**：E8 剧本阶段把 endhook 从 outline 原稿 `"I know what I have to do" + 握拳走出书房` 升级到 `"Whatever it takes." + Gonzalez 卡 + HALLWAY →`，但 `分集大纲.md` L253 没同步。下一集 E9 生成 agent 读陈旧 outline → 元数据 P0 污染 8 处，修复需要双端 replace_all。
> 3. **修源忘 bump version**：主 agent 修源后忘记同步 YAML frontmatter `version: 1 → 2`，导致 `/分镜脚本` 阶段 `source_episode_version` 锁失效，下游 agent 读到陈旧内容无法识别。

### 硬规则

1. **任何修源操作必须外置算术验证**（P0）：
   - 涉及时长字段的修源（场次时长调整、集信息头重写）必须用 `python3 -c "print(...)"` 或 Bash 算术做外置加法验证，并把验证命令 + 输出贴在 `质检检查点.md` 修复记录中作为证据
   - 严禁 "肉眼看起来对了" / "大概闭合" / "应该 OK" 这类口头断言
   - 外置算术命令必须可被他人复跑（保留原样）

2. **outline vs script 漂移双端同步**（P0）：
   - 剧本阶段对以下字段的修源操作必须当场同步回 `分集大纲.md` 对应行：
     - **结尾钩子 / endhook**（相邻集承接命脉）
     - **开场钩子**（承接上一集命脉）
     - **付费卡点位置 / 类型**
     - **关键锚点台词**（Voice Print 原子）
     - **状态差量的最终结果描述**
   - 同步方式：先用 Grep 定位 `分集大纲.md` 对应集数的 endhook/开场钩/卡点行，再用 Edit 或 replace_all 整行重写
   - 验证：修源后必须再 Grep 一次确认 `分集大纲.md` 中不再残留陈旧文本
   - **跨集传染检查**：outline 漂移会通过"下一集承接上一集 outline"机制传染——修 E(N) 剧本后，必须额外 Grep 下游 3 集范围（E(N+1) ~ E(N+3)）的剧本/大纲是否引用了 E(N) 陈旧内容，若有则一并修
   - 未同步 = `/剧本质检` 下一批次必然捕获元数据 P0，成本后推 1 批次

3. **修源必 bump version + 状态回滚**（P0）：
   - 任何对 `分集剧本/第NNN集.md` 内容字段的修源（非格式修正）必须同步更新 YAML frontmatter：
     - `version: N → N+1`
     - `updated_at: YYYY-MM-DD` 更新到当天
   - bump 完成后必须用 `grep "^version:" 分集剧本/第NNN集.md` 或 `head -10` 验证 YAML 确实已更新
   - 漂移检查：修源闭环后，`剧本状态.json` `qcStatus.episodes[{range: 该批次, checkType: "script"}].notes` 必须明文记录 `source_episode_version: N→N+1` 供 `/分镜脚本` 阶段锁
   - **下游 source_episode_version 锁**：下一批 `/分镜脚本` spawn 时，prompt 必须写明源剧本 version（从 `grep "^version:"` 确认），分镜 YAML frontmatter 的 `source_episode_version` 字段必须匹配；不匹配 = P0
   - 常见踩坑：修完剧本内容忘 bump version，分镜阶段锁失效——主 agent 必须在修源任务列表最后一条写 "bump version + 验证 grep"，不得省略
   - 状态回滚：修源开始后，受影响 `checkType="script"` 记录保持 `需修改` 或回退为 `未检查`，直到复检通过；若已有 `checkType="full-read"` 覆盖全剧，必须回退为 `未检查`；若该集已有分镜，必须把对应 `qcStatus.storyboards[*]` 回退为 `未检查` 或 `需修改`；若已跑 `/总检` 或 `/合规`，必须把 `qcStatus.production`、`qcStatus.compliance` 回退为 `未检查`

4. **修源执行顺序固定**（P0）：
   - 顺序：`登记问题与状态回滚 → 源剧本修源 → bump version → grep 验证 → outline 双端同步 → 下游传染检查 → 写质检检查点.md 修复日志 → 重跑受影响剧本质检`
   - **禁止乱序**：常见错误是"先 bump version 再修内容"（bump 时内容还没改，bump 了个寂寞）或 "只修剧本不修 outline"（漂移埋雷）
   - 每一步都必须有 Bash/Grep 输出作为证据贴入 `质检检查点.md`

### 违反示例 vs 合规示例

**违反示例**（E7-E9 修复前一部分动作）：
```
Edit 第008集.md 把 "My father built this. I sign it." → "He built this. I protect this. Same thing."
✓ 完成
```
问题：忘 bump version（仍是 1）+ 没 grep 验证 + 没检查 E9 是否引用 E8 旧台词 + 没写质检日志。

**合规示例**：
```bash
# Step 1: 修源
Edit 第008集.md: "My father built this. I sign it." → "He built this. I protect this. Same thing."
# Step 2: bump version
Edit 第008集.md: version: 1 → 2, updated_at: 2026-04-12
# Step 3: grep 验证
grep "^version:" 分集剧本/第008集.md  # → version: 2 ✓
grep "my father" 分集剧本/第008集.md  # → 0 hits ✓
# Step 4: outline 双端同步
grep -n "My father built" 分集大纲.md  # 若有命中则 Edit
# Step 5: 下游传染检查
grep -rn "My father built this" 分集剧本/第009集.md 分集剧本/第010集.md  # 确认无污染
# Step 6: 写修复日志
Edit 质检检查点.md 追加 QC-SC-E8-001 修复记录（含上述 5 步输出）
```

### 与 §阻塞条件 的关系

- 本节 "修源闭环" 硬规则是 `/剧本质检` 修源后的必走动作，与 §阻塞条件 的判定不冲突——阻塞条件决定 "是否要修"，本节决定 "修的时候要做什么"
- 任一条违反都会在下一批次质检时通过元数据漂移 / version 不锁 / outline 残留重新暴露，成本后推。owner 意识是 **一次修对，不留二次成本**
