## 核心原则

`/总检` 只负责全局收口，不能替代过程质检。

过程质检分成两层：

1. 内建自检：每个生产步骤输出末尾都要自查；`/开始` 与 `/立项` 两个方向锁定步骤只走内建自检，不单独暴露独立质检命令
2. 阶段闸门质检：从 `/设定质检` 开始，通过独立质检命令决定能否进入下一步

## 一、质检时机

| 阶段 | 内建自检 | 独立质检命令触发条件 |
|------|---------|------------------|
| `/开始` 后 | 必须追加；结果写 `选题分析.md` 推荐版末尾，**不落盘** `质检检查点.md` | 无独立命令；P0 回退 `/开始` 重做 |
| `/立项` 后 | 必须追加；结果写 `选题分析.md` 正式版末尾，**不落盘** `质检检查点.md` | 无独立命令；不暴露 `/立项质检`，P0 回退自检 |
| `/设定` 后 | 必须追加 | 角色多 / 关系复杂 / 非现实世界规则 → 必须 `/设定质检`；含超自然/奇幻/能力系统（Werewolf/Vampire/Supernatural/Chosen One/Magic 等）→ **必须**，且六项能力边界缺任一 = P0（定义见 `references/world-character-scene.md` §能力边界设计框架） |
| `/结构` 后 | 必须追加 | 多线推进 / 重反转 / 多卡点 → 必须 `/结构质检` |
| `/分集大纲` 后 | 必须追加 | 总集数 > 30 或前中后段节奏差异明显 → 必须 `/大纲质检` |
| `/分集剧本` 批次 | 每 3-5 集追加一次 | 每批跑 `/剧本质检 {起止集}`；关键卡点/身份揭露/感情翻页集写完后再跑 `/衔接质检 {起止集}` |
| `/分镜脚本` 批次 | 每 3-5 集追加一次 | 每批跑 `/分镜质检 {起止集}`；首个强卡点/重大揭露/高潮反杀/尾钩强切集必须单独抽查 |

> `质检检查点.md` 首次落盘点延后到 `/设定质检` 或任一独立质检命令首次执行时。`/开始` `/立项` 自检结果只写入 `选题分析.md`。

## 一·附、内建自检结果持久化

> `/开始` `/立项` 的自检结果写入 `选题分析.md`。其余所有阶段（`/设定` `/结构` `/分集大纲` `/分集剧本` `/分镜脚本`）的内建自检结果必须追加到 `质检检查点.md`（若已存在），标记为 `内部自检 · {命令名} · {日期} · {状态}`，提供审计追溯。若 `质检检查点.md` 尚未创建（首次独立质检前），自检结果写入命令输出即可，不强制预建。

## 二、内建自检格式

每个生产命令结尾追加：

```markdown
## 内建自检
- 本阶段最可能的 3 个风险：
- 关键专业角色会诊（只写本阶段最 relevant 的 2-4 个）：
  - 市场/题材：
  - 编剧/Showrunner：
  - 导演/画面：
  - 抽卡师/角色运营：
  - 制片/成本：
  - 本地化/合规：
- 当前是否存在 `P0`：
- 若继续推进，最容易崩在哪：
- 建议立即执行的下一条质检命令：
```

说明：

- 不要求每一步把 6 个角色全写一遍，只写当前阶段真正影响质量判断的 2-4 个视角
- 若某一视角给出明确 `P0`，即使其他视角都通过，也不得继续推进
- “抽卡师/角色运营”默认检查角色是否有记忆点、关系站位和追更理由，不是检查美术设定是否华丽

## 三、阶段闸门规则

### `/开始`（内建自检）

> 本段不是独立命令，而是 `/开始` 输出末尾必须追加的自检项；结果写入 `选题分析.md` 推荐版末尾，不单独落盘 `质检检查点.md`。

重点检查：

- 候选方向是否真的适合 TikTok
- 主推方向与备选/变体是否区分清楚
- 主情绪是否清晰
- 前 3 集承诺是否成立
- 方向是否已经被有效收窄
- 用户原始想法是否已完成 TikTok 适配修正

阻塞条件（任一命中 = P0，不得进入 `/立项`）：

- 方向仍靠解释成立
- 主情绪不明确
- 前 3 集回报不成立
- 候选方案彼此区分度太低
- 仍在开放式发散，无法进入正式立项
- 用户尚未明确选择方向，但文档已经写成单一路线或已确认配置

### `/立项`（内建自检）

> 本段不是独立命令，而是 `/立项` 输出末尾必须追加的自检项；结果写入 `选题分析.md` 正式版末尾，不单独落盘 `质检检查点.md`。

重点检查：

- premise 是否 3 秒可懂
- 平台适配是否成立
- 情绪承诺是否清晰
- 风险项是否可控
- 若为出海模式，题材核心是否已完成欧美语境改写

阻塞条件（任一命中 = P0，不得进入 `/设定`）：

- 看点靠解释成立
- 平台不适配
- 主情绪不明确
- premise 高风险不可改写
- 出海模式下，核心驱动仍依赖孝道绑架、宗族秩序、体制权力、地方梗或中文社会背景知识

## 三·附、项目恢复规则（Partial State Recovery）

> 当 `剧本状态.json` 存在但部分 `qcStatus` 字段为 null 或已过时时，按以下规则恢复：

1. **`qcStatus` 全为 null 但 `currentStep` ≠ null**：视为"尚未执行任何质检"。操作者必须按管线顺序从最早的未通过阶段开始补跑质检（`/设定质检` → `/结构质检` → `/大纲质检` → `/剧本质检` → ...），直到所有已完成产物的 `qcStatus` 归位
2. **`质检检查点.md` 被删除但 `qcStatus` 有记录**：`qcStatus` 仍有效（来自 JSON），但需重建 `质检检查点.md`。下次质检时会自动创建新文件
3. **`currentStep` 与 `qcStatus` 矛盾**（如 `currentStep = 分镜脚本` 但 `qcStatus.episodes` 全为 `未检查`）：以 `qcStatus` 为准回退 `currentStep`。操作者不得在 `qcStatus` 未通过的情况下强行推进
4. **跨会话恢复**：使用 `/复盘` 或 `/继续` 时，按 `state-schema.md` §滚动生产的 currentStep 语义 全量判定 `currentStep`

## 三·附二、批次合并上限

- `/剧本质检` 单次最大覆盖 **10 集**。若剧本批次为 1-20 集，必须拆分为 `/剧本质检 1-10` + `/剧本质检 11-20`（盲测样本量和评审深度在 10 集以上会显著衰减）
- `/分镜质检` 单次最大覆盖 **10 集**，理由同上
- 例外：首次全覆盖审查（如 `/总检` 前的摸底）可以超过 10 集，但必须在质检报告中标注"概览级"

## 四、批次质检规则

`/分集剧本` 不要一口气写到底。

推荐批次：

- 10-20 集项目：每 2-3 集一检
- 20-40 集项目：每 3-5 集一检
- 40 集以上项目：每 4-6 集一检

以下情况无论批次大小，都要立即质检：

- 首个强卡点写完
- 重大身份揭露写完
- 感情线翻页写完
- 大纲刚修改过

### 长剧（40集以上）质检分级

40集以上项目面临质检工作量过大的问题，采用以下分级策略：

**A级：必须执行完整质检（`/剧本质检` + `/衔接质检`）**
- 首个强卡点前后各1集
- 重大身份揭露集前后各1集
- 感情线翻页集（关键关系变化）
- 大纲刚修改过、受影响的第一集

**B级：仅执行 `/剧本质检`，跳过 `/衔接质检`**
- 正常推进集（无重大事件）
- 过渡集、铺垫集
- 上一集已做 A级 质检的紧邻集

**C级：仅做内建自检，跳过独立质检命令**
- 与前一批次风格完全一致的连续平稳集
- 仅在集中批量写完后，再统一做一次 `/剧本质检`

分级判定原则：
- 若不确定一集属于哪级，默认 B级
- C级使用上限：连续 C级 不超过 3 集，之后必须至少 B级 一次
- 任何 C级 集写完如发现 P0，立即升级为 A级 处理

### 长剧（40集以上）分镜质检分级

40集以上项目的 `/分镜质检` 也采用分级：

**A级：必须执行完整 `/分镜质检`**
- 首个强卡点集
- 重大身份揭露集
- 高潮反杀集
- 结尾强钩子集
- 新角色首次承担关键戏剧功能的集数

**B级：执行标准 `/分镜质检`，但可只抽查关键镜头**
- 正常推进集
- 铺垫集
- 与上一批次镜头语言基本一致的稳定集

抽查最低要求：
- 开场钩子镜头 1 个
- 本集最大情绪翻点镜头 1 个
- 尾钩镜头 1 个

**C级：仅做分镜内建自检，不单独开质检命令**
- 连续平稳过渡集
- 上一集已做 A级 且本集镜头语言高度继承的紧邻集

使用上限：
- 连续 C级 不超过 2 集
- 任一 C级 集若出现 prompt 漏项、连续性漂移或时长失控，立即升级为 A级

## 五、记录要求

所有阶段质检结果都要追加到 `质检检查点.md`，优先直接套用 `references/templates/qc-checkpoint.md` 的固定模板。

记录时必须满足：

- 每个问题有唯一 ID
- 每个问题有状态
- 每个问题有必改时点
- 修复后要更新原问题状态，而不是只在后面补一句“已修”
- 修复时要按 `references/document-sync.md` 同步更新相关文档
- 质检结论同步更新到 `剧本状态.json` 的 `lastQcStep` 和对应 `qcStatus`
- `/剧本质检 {起止集}` 与 `/衔接质检 {起止集}` 还要同步更新 `qcStatus.episodes`
  - `range` 写实际质检覆盖的集数区间，如 `001-005`、`006-010`，不要固定写死
  - `checkType` 分别写 `script` 或 `continuity`
  - `status` 写 `已通过` 或 `需修改`
  - `blockingIssues` 写当前仍阻塞推进的 QC ID 列表
  - 同一 `range + checkType` 组合更新原记录，不重复追加
- `/分镜质检 {起止集}` 还要同步更新 `qcStatus.storyboards`
  - `range` 写实际质检覆盖的集数区间，如 `001-005`
  - `status` 写 `已通过` 或 `需修改`
  - `blockingIssues` 写当前仍阻塞推进的 QC ID 列表
  - 同一 `range` 组合更新原记录，不重复追加

建议格式：

```markdown
## 2026-03-12 /结构质检
- 检查对象：
- 结论：通过 / 需修改 / 阻塞
- P0：
- P1：
- P2：
- 必改项：
- 下一步：
```

## 六、推进规则

> **权威边界**：本节只讲跨阶段的通用推进原则。具体某一质检命令"不通过时优先修哪里"已下沉到对应 `qc-*.md §执行要求`；问题修复后的"联动字段回填"以 `references/document-sync.md` 为唯一权威源。

1. 阶段内建自检发现 `P0`，先修再质检
2. 阶段闸门质检阻塞时，不得推进下一步
3. `/剧本质检` 不通过时，不得继续写后续批次（具体修复优先级见 `qc-episode.md`）
4. `/衔接质检` `/分镜质检` 不通过时的修复优先级，分别见 `qc-link.md` `qc-storyboard.md`
5. 修复完成后必须按 `document-sync.md` 同步联动字段
